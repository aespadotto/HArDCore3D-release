19 // TetGen computes Delaunay tetrahedralizations, constrained Delaunay tetra- //
20 // hedralizations, and quality Delaunay tetrahedral meshes. The latter are //
21 // nicely graded and whose tetrahedra have radius-edge ratio bounded. Such //
22 // meshes are suitable for finite element and finite volume methods. //
23 // //
24 // TetGen incorporates a suit of geometrical and mesh generation algorithms. //
25 // A brief description of algorithms used in TetGen is found in the first //
26 // section of the user's manual. References are given for users who are //
27 // interesting in these approaches. The main references are given below: //
28 // //
29 // The efficient Delaunay tetrahedralization algorithm is: H. Edelsbrunner //
30 // and N. R. Shah, "Incremental Topological Flipping Works for Regular //
31 // Triangulations". Algorithmica 15: 223-241, 1996. //
32 // //
33 // The constrained Delaunay tetrahedralization algorithm is described in: //
34 // H. Si and K. Gaertner, "Meshing Piecewise Linear Complexs by Constrain- //
35 // ed Delaunay Tetrahedralizations". To appear in Proceedings of the 14th //
36 // International Mesh Roundtable. September 2005. //
37 // //
38 // The Delaunay refinement algorithm is from: J. R. Shewchuk, "Tetrahedral //
39 // Mesh Generation by Delaunay Refinement". Proceedings of the 14th Annual //
40 // Symposium on Computational Geometry, pages 86-95, 1998. //
41 // //
42 // The mesh data structure of TetGen is a combination of two types of mesh //
43 // data structures. The tetrahedron-based mesh data structure introduced //
44 // by Shewchuk is eligible to implement algorithms of generating Delaunay //
45 // tetrahedralizations. However, constrained Delaunay tetrahedralization //
46 // and quality mesh generation algorithms require other mesh elements //
47 // (subfaces, subsegments) be handled at the same time. The triangle-edge //
48 // data structure from Muecke is used for this purpose. These data types //
49 // are handled through a set of fast mesh manipulation primitives. //

493 // //
494 // Geometric predicates //
495 // //
496 // Return one of the values +1, 0, and -1 on basic geometric questions such //
497 // as the orientation of point sets, in-circle, and in-sphere tests. They //
498 // are basic units for the composition of geometric algorithms. TetGen uses //
499 // two 3D geometric predicates, which are the orientation test and the in- //
500 // sphere test (e.g. the locally Deklaunay test). //
501 // //
502 // Orientation test: let a, b, c be a sequence of 3 non-collinear points in //
503 // R^3. They defines a unique hypeplane H. Let H+ and H- be the two spaces //
504 // separated by H, which are defined as follows (using the left-hand rule): //
505 // make a fist using your left hand in such a way that your fingers follow //
506 // the order of a, b and c, then your thumb is pointing to H+. Given any //
507 // point d in R^3, the orientation test returns +1 if d lies in H+, -1 if d //
508 // lies in H-, or 0 if d lies on H. //
509 // //
510 // In-sphere test: let a, b, c, d be 4 non-coplanar points in R^3. They //
511 // defines a unique circumsphere S. Given any point e in R^3, the in-sphere //
512 // test returns +1 if e lies inside S, or -1 if e lies outside S, or 0 if e //
513 // lies on S. //
514 // //
515 // The correctness of geometric predicates is crucial for the control flow //
516 // and hence for the correctness and robustness of an implementation of a //
517 // geometric algorithm. The following routines use arbitrary precision //
518 // floating-point arithmetic. They are fast and robust. It is provided by J. //
519 // Schewchuk in public domain (http://www.cs.cmu.edu/~quake/robust.html). //
520 // The source code are found in a separate file "predicates.cxx". //
521 // //

626 // //
627 // The basic mesh element data structures //
628 // //
629 // There are four types of mesh elements: tetrahedra, subfaces, subsegments, //
630 // and points, where subfaces and subsegments are triangles and edges which //
631 // appear on boundaries. A tetrahedralization of a 3D point set comprises //
632 // tetrahedra and points; a surface mesh of a 3D domain comprises subfaces //
633 // subsegments and points. The elements of all the four types consist of a //
634 // tetrahedral mesh of a 3D domain. However, TetGen uses three data types: //
635 // 'tetrahedron', 'shellface', and 'point'. A 'tetrahedron' is a tetrahedron;//
636 // while a 'shellface' can be either a subface or a subsegment; and a 'point'//
637 // is a point. Theese three data types, linked by pointers comprise a mesh. //
638 // //
639 // A tetrahedron primarily consists of a list of 4 pointers to its corners, //
640 // a list of 4 pointers to its adjoining tetrahedra, a list of 4 pointers to //
641 // its adjoining subfaces (when subfaces are needed). Optinoally, (depending //
642 // on the selected switches), it may contain an arbitrary number of user- //
643 // defined floating-point attributes, an optional maximum volume constraint //
644 // (for -a switch), and a pointer to a list of high-order nodes (-o2 switch).//
645 // Since the size of a tetrahedron is not determined until running time, it //
646 // is not simply declared as a structure. //
647 // //
648 // The data structure of tetrahedron also stores the geometrical information.//
649 // Let t be a tetrahedron, v0, v1, v2, and v3 be the 4 nodes corresponding //
650 // to the order of their storage in t. v3 always has a negative orientation //
651 // with respect to v0, v1, v2 (ie,, v3 lies above the oriented plane passes //
652 // through v0, v1, v2). Let the 4 faces of t be f0, f1, f2, and f3. Vertices //
653 // of each face are stipulated as follows: f0 (v0, v1, v2), f1 (v0, v3, v1), //
654 // f2 (v1, v3, v2), f3 (v2, v3, v0). //
655 // //
656 // A subface has 3 pointers to vertices, 3 pointers to adjoining subfaces, 3 //
657 // pointers to adjoining subsegments, 2 pointers to adjoining tetrahedra, a //
658 // boundary marker(an integer). Like a tetrahedron, the pointers to vertices,//
659 // subfaces, and subsegments are ordered in a way that indicates their geom- //
660 // etric relation. Let s be a subface, v0, v1 and v2 be the 3 nodes corres- //
661 // ponding to the order of their storage in s, e0, e1 and e2 be the 3 edges,//
662 // then we have: e0 (v0, v1), e1 (v1, v2), e2 (v2, v0). //
663 // //
664 // A subsegment has exactly the same data fields as a subface has, but only //
665 // uses some of them. It has 2 pointers to its endpoints, 2 pointers to its //
666 // adjoining (and collinear) subsegments, a pointer to a subface containing //
667 // it (there may exist any number of subfaces having it, choose one of them //
668 // arbitrarily). The geometric relation between its endpoints and adjoining //
669 // subsegments is kept with respect to the storing order of its endpoints. //
670 // //
671 // The data structure of point is relatively simple. A point is a list of //
672 // floating-point numbers, starting with the x, y, and z coords, followed by //
673 // an arbitrary number of optional user-defined floating-point attributes, //
674 // an integer boundary marker, an integer for the point type, and a pointer //
675 // to a tetrahedron (used for speeding up point location). //
676 // //
677 // For a tetrahedron on a boundary (or a hull) of the mesh, some or all of //
678 // the adjoining tetrahedra may not be present. For an interior tetrahedron, //
679 // often no neighboring subfaces are present, Such absent tetrahedra and //
680 // subfaces are never represented by the NULL pointers; they are represented //
681 // by two special records: `dummytet', the tetrahedron fills "outer space", //
682 // and `dummysh', the vacuous subfaces which are omnipresent. //
683 // //
684 // Tetrahedra and adjoining subfaces are glued together through the pointers //
685 // saved in each data fields of them. Subfaces and adjoining subsegments are //
686 // connected in the same fashion. However, there are no pointers directly //
687 // gluing tetrahedra and adjoining subsegments. For the purpose of saving //
688 // space, the connections between tetrahedra and subsegments are entirely //
689 // mediated through subfaces. The following part explains how subfaces are //
690 // connected in TetGen. //
691 // //
693 
695 // //
696 // The subface-subface and subface-subsegment connections //
697 // //
698 // Adjoining subfaces sharing a common edge are connected in such a way that //
699 // they form a face ring around the edge. It is in deed a single linked list //
700 // which is cyclic, e.g., one can start from any subface in it and traverse //
701 // back. When the edge is not a subsegment, the ring only has two coplanar //
702 // subfaces which are pointing to each other. Otherwise, the face ring may //
703 // have any number of subfaces (and are not all coplanar). //
704 // //
705 // How is the face ring formed? Let s be a subsegment, f is one of subfaces //
706 // containing s as an edge. The direction of s is stipulated from its first //
707 // endpoint to its second (according to their storage in s). Once the dir of //
708 // s is determined, the other two edges of f are oriented to follow this dir.//
709 // The "directional normal" N_f is a vector formed from any point in f and a //
710 // points orthogonally above f. //
711 // //
712 // The face ring of s is a cyclic ordered set of subfaces containing s, i.e.,//
713 // F(s) = {f1, f2, ..., fn}, n >= 1. Where the order is defined as follows: //
714 // let fi, fj be two faces in F(s), the "normal-angle", NAngle(i,j) (range //
715 // from 0 to 360 degree) is the angle between the N_fi and N_fj; then fi is //
716 // in front of fj (or symbolically, fi < fj) if there exists another fk in //
717 // F(s), and NAangle(k, i) < NAngle(k, j). The face ring of s is: f1 < f2 < //
718 // ... < fn < f1. //
719 // //
720 // The easiest way to imagine how a face ring is formed is to use the right- //
721 // hand rule. Make a fist using your right hand with the thumb pointing to //
722 // the direction of the subsegment. The face ring is connected following the //
723 // direction of your fingers. //
724 // //
725 // The subface and subsegment are also connected through pointers stored in //
726 // their own data fields. Every subface has a pointer ti its adjoining sub- //
727 // segment. However, a subsegment only has one pointer to a subface which is //
728 // containing it. Such subface can be choosn arbitrarily, other subfaces are //
729 // found through the face ring. //
730 // //
732

770 // The mesh handle (triface, face) data types //
771 // //
772 // Two special data types, 'triface' and 'face' are defined for maintaining //
773 // and updating meshes. They are like pointers (or handles), which allow you //
774 // to hold one particular part of the mesh, i.e., a tetrahedron, a triangle, //
775 // an edge and a vertex. However, these data types do not themselves store //
776 // any part of the mesh. The mesh is made of the data types defined above. //
777 // //
778 // Muecke's "triangle-edge" data structure is the prototype for these data //
779 // types. It allows a universal representation for every tetrahedron, //
780 // triangle, edge and vertex. For understanding the following descriptions //
781 // of these handle data structures, readers are required to read both the //
782 // introduction and implementation detail of "triangle-edge" data structure //
783 // in Muecke's thesis. //
784 // //
785 // A 'triface' represents a face of a tetrahedron and an oriented edge of //
786 // the face simultaneously. It has a pointer 'tet' to a tetrahedron, an //
787 // integer 'loc' (range from 0 to 3) as the face index, and an integer 'ver' //
788 // (range from 0 to 5) as the edge version. A face of the tetrahedron can be //
789 // uniquly determined by the pair (tet, loc), and an oriented edge of this //
790 // face can be uniquly determined by the triple (tet, loc, ver). Therefore, //
791 // different usages of one triface are possible. If we only use the pair //
792 // (tet, loc), it refers to a face, and if we add the 'ver' additionally to //
793 // the pair, it is an oriented edge of this face. //
794 // //

796 // It has a pointer 'sh' to a subface, an integer 'shver'(range from 0 to 5) //
797 // as the edge version. The pair (sh, shver) determines a unique oriented //
798 // edge of this subface. A 'face' is also used to represent a subsegment, //
799 // in this case, 'sh' points to the subsegment, and 'shver' indicates the //
800 // one of two orientations of this subsegment, hence, it only can be 0 or 1. //
801 // //
802 // Mesh navigation and updating are accomplished through a set of mesh //
803 // manipulation primitives which operate on trifaces and faces. They are //

1540 // //
1541 // Mesh Local Transformation Operators //
1542 // //
1543 // These operators (including flips, insert & remove vertices and so on) are //
1544 // used to transform (or replace) a set of mesh elements into another set of //
1545 // mesh elements. //
1546 // //
1547 // Flip Types //
1548 // //
1549 // If abc is a hull face, it is unflipable, and is locally Delaunay. In the //
1550 // following, we assume abc is an interior face, and the other tetrahedron //
1551 // adjoining at abc is bace. //
1552 // //
1553 // If the convex hull CH of the set {a, b, c, d, e} only has four vertices, //
1554 // i.e., one vertex lies inside CH, then abc is unflipable, and is locally //
1555 // Delaunay. If CH is the vertex set itself, we have the following cases to //
1556 // determine whether abc is flipable or not. //
1557 // //
1558 // If no four points of {a, b, c, d, e} are coplanar, a 2-to-3 flip can be //
1559 // applied to abc if the edge de crosses the triangle abc; a 3-to-2 flip can //
1560 // be applied to abc if ab crosses cde, and abde exists, otherwise, face abc //
1561 // is unflipable, i.e., the tetrahedron abde is not present. //
1562 // //
1563 // If four points of {a, b, c, d, e} are coplanar (two faces are coplanar). //
1564 // Assume faces abd and abe are coplanar (it is impossible be abc). If a, b, //
1565 // d, e form a non-convex quadrilateral, then abc is unflipable, furthermore,//
1566 // it is locally Delaunay. Assume they are convex quadrilateral, if abd and //
1567 // abe are hull faces, a 2-to-2 flip can be applied to abc; if abd and abe //
1568 // are interior faces, assume two tetrahedra adjoining abd and abe at the //
1569 // opposite sides are abdg and abef, respectively. If g = f, a 4-to-4 flip //
1570 // can be applied to abc, otherwise, abc is unflipable. //
1571 // //
1572 // There are other cases which can cause abc unflipable. If abc is a subface,//
1573 // a 2-to-3 flip is forbidden; if ab is a subsegment, flips 3-to-2, 2-to-2, //
1574 // and 4-to-4 are forbidden. //
1575 // //
